# Provenance Guard

A backend that classifies whether submitted text was written by a human or generated by AI, scores its confidence, surfaces a plain-language transparency label, and handles creator appeals.

## How it works

```
POST /submit → validate → rate limit → Signal A (stylometric) + Signal B (Groq LLM judge)
            → fuse into one confidence (0–1) → map to a transparency label → write audit record → respond
```

Each submission runs through **two independent detectors that fail in different
ways**, so their agreement is meaningful: **Signal A** (deterministic stylometry —
burstiness, lexical diversity, repetition; pure stdlib) and **Signal B** (a
Groq-hosted LLM judge that reads meaning and register). The **scorer** fuses them
into one confidence score (Signal-B-primary; see below), the **label mapper**
turns that score into one of three plain-language labels, and every decision is
written to an **audit log** that doubles as the record store the appeal flow later
mutates. Full architecture narrative + diagram live in [`planning.md`](planning.md).

**Endpoints:** `POST /submit` (classify), `POST /appeal` (contest a verdict),
`GET /content/{id}` (one record), `GET /log` (audit log; `?status=under_review`
is the human appeal queue).

### Running it

```bash
pip install -r requirements.txt          # flask, flask-limiter, groq, python-dotenv
echo "GROQ_API_KEY=gsk_..." > .env        # Signal B reads this; without it the judge
                                          # degrades to neutral 0.5 and fusion uses Signal A alone
python3 app.py                           # serves on http://localhost:5001
```

```bash
# classify a submission (body kept in a file so apostrophes/quotes need no escaping)
curl -s -X POST http://localhost:5001/submit -H "Content-Type: application/json" \
  -d @rl_payload.json | python3 -m json.tool
```

## Note: why one signal isn't enough (empirical finding)

While testing Signal A (the stylometric signal) in isolation, a clearly AI-generated
paragraph was misclassified as **human**:

> "Artificial intelligence represents a transformative paradigm shift in modern society.
> It is important to note that while the benefits of AI are numerous, it is equally
> essential to consider the ethical implications. Furthermore, stakeholders across
> various sectors must collaborate to ensure responsible deployment."

Signal A scored this **0.247 → human**, because the text has varied sentence lengths,
rich vocabulary, and no repetition — the surface statistics all look human.

**Conclusion:** surface statistics (burstiness, lexical diversity, repetition) detect
_clumsy_ AI (e.g. "The product is very good. The product is very nice…" → 0.889) but are
**blind to fluent AI**. A human recognizes the paragraph above as AI from its hollow,
hedging register — which is _semantic_, not statistical. No amount of additional
stylometric features would fix this; they would all fail on the same input.

This is the concrete motivation for **Signal B (a semantic LLM judge)**: it fails on
_different_ inputs than Signal A, which is exactly what makes combining the two
(fusion) more reliable than either alone. A single signal of any kind is unreliable —
that is the core reason this system uses two.

## Design change made during development: weighting the Groq layer more heavily

The original plan fused the two signals as near-equal peers (Signal A 0.4, Signal B 0.6)
with a _disagreement penalty_ that pushed the score toward "uncertain" whenever the two
conflicted. Testing the batch above broke that design: because stylometric cues like
**sentence structure are not a guaranteed indicator that something was written by AI**,
Signal A confidently mislabels fluent AI as human — and the equal-peer fusion then let
that wrong Signal A _cancel out_ a correct Signal B, collapsing the verdict to "uncertain."

So during development I rebalanced the fusion to **weight the Groq (Signal B) layer much
more heavily**:

- Signal B weight raised to **0.8**, Signal A dropped to **0.2** (Groq is the primary
  detector; the stylometric stats only nudge).
- The disagreement penalty was removed — a blind statistical signal should not be able to
  veto the semantic judge.
- The AI threshold was lowered from 0.85 to **0.75**, since protection against false
  accusations now comes from Groq's reliability (it also correctly recognizes genuine
  human writing) rather than from an extreme cutoff.

**Result — same paragraph, before vs. after:**

|                              | Signal A | Signal B | Fused confidence | Verdict   |
| ---------------------------- | -------- | -------- | ---------------- | --------- |
| Before (Signal A alone)      | 0.247    | —        | 0.247            | human ❌  |
| After (Groq-weighted fusion) | 0.25     | 0.90     | **0.769**        | **ai** ✅ |

And a genuine human poem (clean, spare style that fools the statistics) still lands at
~0.41 → **uncertain**, _not_ falsely accused — confirming that weighting Groq more heavily
catches fluent AI without sacrificing protection for real writers.

## Confidence scoring in action — example submissions

The score is **not a constant** — it moves with the evidence. Three real runs through the
pipeline, deliberately spanning the confidence bands:

**High-confidence AI — formulaic, hedging prose (~56 words):**

> "I understand that managing your time can be challenging. First, it is important to
> prioritize your tasks effectively. Additionally, breaking large projects into smaller,
> manageable steps can significantly enhance productivity. Furthermore, maintaining a
> consistent schedule is essential for long-term success. Ultimately, by following these
> strategies, you can achieve a healthy work-life balance and reach your goals."

| Signal A | Signal B | Fused confidence | Verdict                    |
| -------- | -------- | ---------------- | -------------------------- |
| 0.319    | 0.98     | **0.848**        | **ai** (≈85% confident AI) |

Signal B nails it (0.98) on the AI tells — the "I understand that…" opener, the
First/Additionally/Furthermore/Ultimately scaffolding, generic advice with no specific
detail. Its rationale: _"formulaic scaffolding and a smoothly balanced corporate tone."_
Note Signal A is only **0.319** (the prose is smooth but not statistically extreme) — yet
because Signal B is primary, the fused score still clears the 0.75 AI bar. This is the
B-primary design doing exactly what it was rebalanced to do: let the semantic judge carry a
verdict the statistics alone would have missed.

**High-confidence human — a personal gaming intro (~97 words):**

> "Hi, my name is Kyle Li. Nice to meet you. I'm a big fan of video games like Team
> Fortress 2, Minecraft, and Grand Theft Auto. Honestly, I've sunk way too many hours
> into TF2 over the years… How about you? Got any games you keep coming back to?"

| Signal A | Signal B | Fused confidence | Verdict                         |
| -------- | -------- | ---------------- | ------------------------------- |
| 0.035    | 0.10     | **0.087**        | **human** (≈91% confident human) |

Both signals lean human, and the text is long enough that the length cap doesn't damp
it, so it lands well under the 0.25 human bar. Signal B's rationale: _"contains specific
personal details and idiosyncratic preferences."_

**Lower-confidence case — a short, spare vignette (~29 words):**

> "The sun dipped below the horizon, painting the sky in hues of amber and rose. I sat on
> the porch, coffee in hand, watching the neighborhood slowly go quiet."

| Signal A | Signal B | Fused confidence | Verdict       |
| -------- | -------- | ---------------- | ------------- |
| 0.475    | 0.30     | **0.38**         | **uncertain** |

The signals disagree mildly and the text is short, so the length cap pulls the score
toward 0.5 — it lands in the uncertain band rather than committing.

Three inputs, three bands — **0.848** (AI), **0.087** (human), **0.38** (uncertain). The
scoring produces meaningful variation, not a fixed answer.

**What I'd change for real deployment.** The weights and thresholds (0.2/0.8,
0.25/0.75, the 40-word length cap) are hand-tuned against a handful of test inputs,
not calibrated. For production I'd fit them against a labeled corpus of known-AI and
known-human text, measure the actual false-accusation rate, and likely make the length
cap a smooth function rather than a hard 40-word line. I'd also swap the in-memory/JSONL
audit log for a real database and add authentication so appeals tie to verified owners.

## Transparency labels — the three variants

A reader never sees a raw score; they see one of three plain-language labels (text below
is verbatim from [`scoring.py`](scoring.py)). `{pct}` is `round(confidence·100)` for the
AI label and `round((1−confidence)·100)` for the human label; the uncertain label shows
no number by design.

**High-confidence AI** — `confidence ≥ 0.75`:

> ⚠️ This content appears to be AI-generated. Our system analyzed its writing patterns and
> is highly confident (about {pct}%) it was produced by an AI tool rather than written by a
> person. This is an automated assessment, not a verdict — the creator can contest it.

**High-confidence human** — `confidence < 0.25`:

> ✓ This content appears to be human-written. Our system found writing patterns consistent
> with a human author and is confident (about {pct}%) it was not AI-generated.

**Uncertain** — `0.25 ≤ confidence < 0.75`:

> ❓ We couldn't confidently determine how this content was created. Our system found mixed
> signals — it may have been written by a person, generated by AI, or a mix of both. We're
> showing this note instead of a definitive label to avoid making a wrong call.

The uncertain label omits a percentage on purpose: surfacing "55%" would imply a precision
the system doesn't have. Plain language carries the uncertainty instead.

## Known limitations (where this fails, and why)

Concrete failure modes tied to **how the signals work** — not "it needs more data":

1. **Spare, low-entropy human writing — minimalist poetry, clipped prose.** Signal A scores
   _burstiness_ (sentence-length variance) and _lexical diversity_. A human who writes in
   short, uniform, controlled sentences produces exactly the low-variance, low-diversity
   texture the heuristic reads as AI — so Signal A leans AI on genuine human work. This is a
   property of the feature itself: no stylometric statistic can distinguish a disciplined
   human voice from a machine. We soften it by weighting Signal A only 0.2 and leaning on
   Signal B (which reads meaning and usually recognizes a real human poem), but a clean,
   confident human poem can still be misjudged — which is exactly why the appeal path exists.
2. **Very short text — a haiku, a caption, a two-sentence excerpt.** Below 15 words / 2
   sentences Signal A returns a flat 0.5, and burstiness/diversity are statistically
   meaningless on ~10–25 words. The length cap then shrinks the fused score toward 0.5, so
   almost anything short lands "uncertain" (see the 29-word vignette above). Honest, but blunt.
3. **Non-English text, source code, or structured lists.** The stylometric heuristics are
   tuned for English prose and misfire on other languages, code, or bulleted data. These
   should be detected and routed separately — out of scope here.

## Rate limiting

The submission endpoint is the expensive path: every `POST /submit` fans out to
two signals, one of which is a **Groq LLM call** (network latency + per-token
cost). Left unprotected, a script could flood it — running up the bill and
starving real users. So Flask-Limiter (keyed by client IP via
`get_remote_address`) guards the write endpoints:

| Endpoint       | Limit                  | Reasoning                                                                                                                                                                                                                                                                                             |
| -------------- | ---------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `POST /submit` | **10/minute; 100/day** | A real writer checks a handful of drafts in a burst — 10/min covers active editing comfortably while blocking a script firing hundreds. 100/day bounds total Groq spend per IP and stops a slow drip that stays under the minute cap. A genuine individual won't submit 100 distinct pieces in a day. |
| `POST /appeal` | **20/hour**            | Appeals are rare by nature (you contest a verdict, you don't spam them) and each one rewrites the audit log. 20/hour is generous for legitimate use yet caps abuse of the mutation path.                                                                                                              |

Read endpoints (`GET /log`, `GET /content/{id}`) are left unlimited — they're
cheap and read-only. Over-limit requests get a **429** as JSON (matching the
rest of the API's error shape), e.g. `{"error": "rate limit exceeded", ...}`.

## Spec reflection: where the plan helped, and where I diverged

**Where the spec guided me.** Writing `planning.md` first gave me a precise contract to
build against. The §2 threshold table and the three verbatim label strings in §3 made the
label mapper a near-mechanical translation — I implemented `to_label()` straight from those
numbers and strings and verified it reproduced them exactly. And the §4 decision to treat
the audit log as **one mutable record per piece of content** (not two separate logs) is what
made the appeal endpoint simple: the appeal just looks the record up and mutates it in place,
exactly as the diagram's shared-node argument predicted.

**Where I diverged.** The original plan fused the two signals as near-equal peers
(`0.4·A + 0.6·B`) with a _disagreement penalty_, and set the AI bar at 0.85. Testing broke
it: a clearly-AI paragraph scored 0.247 on Signal A, and equal-peer fusion let that wrong
signal cancel a correct Signal B, collapsing the verdict to "uncertain."

The deeper reason is _why_ Signal A was wrong, and it's why I gave Groq the heavier weight.
Stylometric features like sentence length and variance only measure surface **texture**, and
texture is fundamentally ambiguous — it misfires in **both directions**: fluent modern AI
writes with varied, human-like rhythm (so the statistics call it human), while some genuine
humans write with machine-like uniformity — spare poetry, minimalist prose (so the statistics
call them AI, the Mara case). The same measurement points the wrong way depending on the
writer, because identical texture can come from either. Groq earns the weight because it reads
**meaning and register** — formulaic scaffolding, hollow hedging, the absence of lived detail —
which is what actually separates fluent AI from a real person. Stylometry reliably catches only
_clumsy_, repetitive AI, so it stays a light corroborator (0.2), not the decider.

So I rebalanced to **Signal-B-primary** (`0.2·A + 0.8·B`), dropped the disagreement penalty,
and lowered the AI threshold to **0.75** — false-accusation protection now comes from Signal
B's reliability rather than an extreme cutoff. (A smaller divergence: the appeal field was
renamed from `reason` to `creator_reasoning` to match the assignment's endpoint contract.)

## AI usage

I built this with an AI coding assistant, feeding it the relevant `planning.md` sections and
verifying each slice before wiring it in. Specific instances:

1. **Generating the appeals workflow + label mapper (M5).** I directed it to generate
   `POST /appeal` and the label mapper from the planning.md appeals/label sections. It
   produced a working endpoint but (a) named the request field `reason`, while the assignment
   required `creator_reasoning`, and (b) correctly flagged that the label function already
   existed from M4 and _verified_ it against the spec instead of duplicating it. I had it
   switch the field to `creator_reasoning` (keeping `reason` as a backward-compatible alias)
   and keep the audit-log-as-record-store mutation it wrote.

2. **Adding rate limiting.** I directed it to apply Flask-Limiter to the submission endpoint
   with defensible limits. It produced an IP-keyed limiter, proposed `10/min; 100/day` for
   `/submit` and `20/hour` for `/appeal` with written reasoning, and added a JSON 429 handler
   so over-limit responses match the rest of the API (which I hadn't asked for, but kept). I
   accepted the limits after checking the reasoning held up.

3. **Debugging a failing test.** When a rate-limit test returned a wall of `000` responses, I
   had it diagnose the cause. It identified two issues — the server wasn't running on the port,
   and apostrophes in the JSON payload were breaking bash's single-quote escaping — and
   recommended moving the body into a file and using `curl -d @file`, which I adopted
   (`rl_payload.json`).
